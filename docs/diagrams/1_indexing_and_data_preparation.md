# Подсистема индексации и подготовки данных

Диаграмма: `docs/diagrams/indexing_and_data_preparation.puml`

## Что отражено

1. Входной путь индексации в проекте:
`POST /v1/documents/bulk` -> внутренняя обработка в приложении.

2. Подготовка данных перед индексацией:
- разбиение входных элементов на чанки по `BULK_INDEX_BATCH_SIZE`;
- построение embedding-текста из доменных `vector_source_fields`.

3. Что делает сервис на bulk-индексации:
- строит embedding text из `vector_source_fields` домена;
- считает эмбеддинги батчем (`embed_texts`), при ошибке переходит на per-item fallback (`embed_text`);
- пишет в OpenSearch через bulk;
- для успешно записанных в OpenSearch документов пишет граф в Neo4j;
- собирает итоговые `indexed`/`failed_ids`.

4. Что делает OpenSearch adapter:
- `ensure_index` (HEAD, при необходимости PUT с доменной конфигурацией индекса);
- `/_bulk?refresh=true`.

5. Что делает Neo4j adapter:
- нормализует row по полям из доменной graph ingest конфигурации;
- запускает рендеренный доменный Cypher-шаблон ingest.

## Ограничение диаграммы

Диаграмма описывает именно внутренний кодовый поток приложения (без внешних вспомогательных скриптов). В ней нет поведения, которого нет в проекте.
