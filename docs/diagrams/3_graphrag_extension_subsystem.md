# Графовое расширение (GraphRAG) в системе

Диаграмма: `docs/diagrams/graphrag_extension_subsystem.puml`

## Как это работает в текущем коде

1. Построение графа при индексации
- При `POST /v1/documents/bulk` сервис после успешной записи документов в OpenSearch вызывает `Neo4jAdapter.ingest_documents(...)`.
- Адаптер нормализует поля по доменной конфигурации (`search.graph.ingest` из `index_config.json`) и выполняет доменный Cypher-шаблон `graph_ingest.cypher.mustache`.
- В граф записываются узлы/связи (фильм, актеры, режиссеры, сценаристы, страны) по доменным label/relation настройкам.

2. Графовое обогащение результатов поиска
- В `SearchService.search(...)` после merge lexical/vector результатов всегда вызывается `Neo4jAdapter.fetch_graph_context(...)`.
- Адаптер выполняет доменный шаблон `graph_context.cypher.mustache` и возвращает контекст по `item_id`.
- Сервис добавляет этот контекст в `payload.graph` каждого релевантного результата:
  - `connections`
  - `related_movies`.

3. Использование граф-контекста в подсистеме LLM
- В `POST /v1/ask` LLM получает уже обогащенные `items`.
- `LLMService` берет `payload.graph`, компактизирует его (ограничивает объем), добавляет в prompt и отправляет в модель.
- Итог: ответ LLM строится с учетом графовых связей, а не только текстового payload.

## Что важно

- GraphRAG в проекте реализован как расширение retrieval-этапа: сначала гибридный поиск, затем графовое обогащение, затем использование обогащенного контекста в LLM.
- При сбое запроса в Neo4j адаптер возвращает пустой контекст, поэтому ответ формируется без графового обогащения только для этого запроса.
- Все графовые детали (Cypher и поля ingest) вынесены в доменные артефакты, а не захардкожены в бизнес-логике.
